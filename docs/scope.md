# 08 · Scope & Milestones

> What needs to be delivered for the project to be playable.

The scope is defined by a practical milestone path, ordered so the **fun is proven before content is poured in**.

## Milestone path

```
Gun System Complete
       │
       ▼
Room System Prototype
       │
       ▼
Core Loop Playable        ◀── the pivot: is it fun yet?
       │
       ▼
     Boss v1
       │
       ▼
   Full Content
       │
       ▼
  Polish & Ship
```

| # | Milestone | State |
|---|---|---|
| 1 | Gun System Complete | <span class="pill wip">In progress</span> |
| 2 | Room System Prototype | <span class="pill wip">In progress</span> |
| 3 | Core Loop Playable | <span class="pill todo">Next</span> |
| 4 | Boss v1 | <span class="pill todo">Planned</span> |
| 5 | Full Content | <span class="pill todo">Planned</span> |
| 6 | Polish & Ship | <span class="pill todo">Planned</span> |

## Development order

1. **Establish a playable core loop first**: it's the foundation everything else hangs on.
2. **Finalize room generation and boss progression early**, before content depends on them.
3. **Add content only after the loop is confirmed engaging**: no building rooms for a loop that isn't fun.
4. **Finish with audio, UI refinement, and performance.**

---

!!! tip "The one rule of the path"
    Don't skip step 3's gate. Content built on an unproven loop is the most expensive thing a small team can waste time on. Prove *Enter / Survive / Collect / Return* is fun with one gun and a handful of rooms first.
